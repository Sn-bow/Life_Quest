## 0. 원칙

1. 정식 배포 판단 기준은 "빌드가 된다"가 아니라 "신규 사용자가 돈과 시간을 써도 납득 가능한 완성도"이다.
2. 작업 착수 전에는 관련 자료를 먼저 조사한다. 최신 제품, 정책, 플랫폼 요구사항, 레퍼런스 게임은 공식/1차 출처를 우선한다.
3. 코드 변경은 작은 단위로 진행하고, 각 단위마다 `flutter analyze`, 관련 테스트, 가능하면 실제 기기/에뮬레이터 확인을 남긴다.
4. 사용자 변경사항과 기존 미커밋 작업은 보존한다. 정리 목적의 광범위 리팩터링은 배포 차단 위험이 명확할 때만 한다.
5. Claude와 Codex는 [SHARED_WORK_LOG.md](./SHARED_WORK_LOG.md)에 모든 작업 내역, 결과, 남은 위험을 기록한다.

## 1. 현재 앱의 배포 위험 진단

### 강점

- Flutter + Firebase 기반의 앱 골격은 이미 갖춰져 있다.
- Android applicationId는 `com.lifequest.app`로 변경되어 있다.
- Soul Deck 카드 207장, 몬스터 31종+보스, 유물 31개, 이벤트 10개, 4개 언어 ARB 구조가 존재한다.
- 몬스터 PNG, 배경 PNG, 일부 카드 프레임 PNG, 유물 PNG, SFX/BGM 일부가 이미 존재한다.
- `flutter analyze`, `flutter test`, release AAB 성공 기록이 기존 문서에 있다.

### 정식 배포 차단급 위험

- 실제 기기 장기 QA 기록이 부족하다. Firebase, AdMob, IAP, 알림, 홈 위젯, Flame 렌더링은 실제 Android 기기 검증이 필요하다.
- 게임 전투 화면 일부가 `CustomPainter`/Canvas 도형 기반이다. 정식 배포용 비주얼로 보기 어렵다.
- 기존 문서 일부가 인코딩 깨짐 상태라 인수인계 품질이 낮다.
- Play Console 수동 작업이 남아 있다: Firebase 패키지 등록, AdMob ID 확인, IAP/Cloud Functions 배포, 데이터 보안, IARC, 스토어 등록.
- 게임 밸런스는 자동 테스트만으로 보장되지 않는다. 실제 플레이 루프별 승률/피로도/보상 체감 검증이 필요하다.

## 2. 컨텐츠 계획

### 2-1. 컨텐츠 인벤토리 감사

대상:

- 퀘스트: 일간/주간/월간/연간, 커스텀 보상, 반복/완료/보상 루프
- 캐릭터: 레벨, 스탯, 장비, 스킬, 업적, 칭호
- Soul Deck: 카드 207장, 덱 편집, 카드팩, 전투 보상, 상점, 휴식, 이벤트, 무한 타워
- 경제: 골드, XP, 카드 포인트, 유물, 장비, 코스메틱, IAP
- 다국어: 한국어/영어/일본어/중국어 문맥 품질

작업:

1. 전체 컨텐츠 테이블을 CSV/Markdown으로 추출한다.
2. 카드별 비용, 피해, 방어, 드로우, 에너지 회복, 상태이상, 희귀도, 업그레이드 차이를 표로 만든다.
3. 몬스터별 HP/공격/의도/보상/챕터 곡선을 표로 만든다.
4. 유물 31개 효과를 "무조건 강함", "조건부", "함정", "중복 시 위험"으로 분류한다.
5. 각 언어 ARB에서 누락, 어색한 직역, UI 길이 초과 위험을 점검한다.

완료 기준:

- `docs/content-audit.md` 생성
- 카드/몬스터/유물/이벤트에 "배포 가능", "수정 필요", "보류" 상태 부여
- 모든 사용자 노출 텍스트가 ARB 또는 명시적 예외로 관리됨

### 2-2. 초반 30분 경험 강화

문제:

- Life Quest는 생활관리 앱과 카드 로그라이크가 결합되어 있어 첫 사용자에게 정보량이 많다.
- 튜토리얼이 약하면 사용자는 퀘스트 앱인지 게임인지 이해하기 전에 이탈한다.

작업:

1. 첫 실행 3단계 온보딩 재검수: 퀘스트 생성, 스탯 성장, Soul Deck 보상 루프를 한 문장씩 명확히 전달한다.
2. 첫 퀘스트 완료 후 카드팩 또는 첫 전투로 자연스럽게 연결되는 "첫 승리 루프"를 설계한다.
3. Soul Deck 첫 진입 시 강제 튜토리얼 전투를 제공한다.
4. 상태이상/에너지/블록/의도 아이콘 용어집을 전투 화면에서 접근 가능하게 한다.

완료 기준:

- 신규 사용자가 10분 안에 퀘스트 생성, 완료, 보상 확인, 첫 전투 시작까지 도달
- 튜토리얼 스킵/다시보기 가능
- 튜토리얼 상태가 저장되어 반복 노출되지 않음

### 2-3. 라이브 서비스 전 콘텐츠 안정화

작업:

1. 출시 버전 컨텐츠 범위를 고정한다. 출시 직전 새 카드/유물 추가 금지.
2. 카드 밸런스 수정은 `balance_changes.md`에 이유와 전후 수치를 남긴다.
3. "일상 RPG" 핵심과 무관한 장식 기능은 정식 배포 이후로 미룬다.
4. 아바타/캐릭터 커스터마이징은 프로젝트 주의사항에 따라 재도입하지 않는다.

## 3. 디자인 계획

### 3-1. 전역 UI 디자인 감사

대상 화면:

- 로그인/회원가입/온보딩/로딩
- 메인/퀘스트/상태/인벤토리/상점/스킬/업적/리포트/타이머/설정
- 던전 홈/맵/전투/이벤트/상점/휴식/결과/카드 컬렉션/카드팩/무한 타워

작업:

1. 각 화면의 스크린샷을 밝은/어두운 테마, 작은 Android 화면, 일반 Android 화면으로 캡처한다.
2. 텍스트 잘림, 버튼 터치 영역, 대비, 반복 컴포넌트, 빈 상태, 오류 상태를 체크한다.
3. 화면별 핵심 행동이 첫 화면에서 보이는지 평가한다.
4. 동일 역할의 카드, 버튼, 뱃지, 바, 아이콘 스타일을 통일한다.

완료 기준:

- `docs/design-audit.md` 생성
- 각 화면별 P0/P1/P2 이슈 목록 작성
- P0: 출시 전 필수, P1: 출시 후보, P2: 출시 후 개선으로 분류

### 3-2. 에셋 우선 디자인 규칙

즉시 적용 규칙:

- 이미 존재하는 PNG 에셋을 우선 사용한다.
- 새 비주얼이 필요하면 이미지 생성 또는 외부 에셋 조사를 먼저 한다.
- 플레이어, 몬스터, 카드 프레임, 카드 일러스트, 전투 배경, 주요 이펙트를 단순 도형만으로 대체하지 않는다.
- `CustomPainter`는 임시 디버그, 마스크, 보조 오버레이, 수치 시각화에만 제한한다.
- 정식 배포 UI에 "그림판 선 긋기"처럼 보이는 도형 에셋을 남기지 않는다.

기록 위치:

- [AI_WORK_RULES.md](./AI_WORK_RULES.md)
- [SHARED_WORK_LOG.md](./SHARED_WORK_LOG.md)

### 3-3. 브랜드/아트 방향

Life Quest의 디자인 핵심:

- 일상 관리 화면: 차분하고 빠르게 읽히는 생산성 앱 톤
- 게임 화면: 카드 로그라이크의 몰입감, 명확한 정보 위계, 손맛 있는 피드백
- 두 영역 연결: 퀘스트 완료가 캐릭터 성장과 카드 보상으로 이어진다는 일관된 시각 언어

금지:

- 생활 관리 화면 전체를 과한 판타지 장식으로 덮는 것
- 게임 화면을 일반 생산성 앱 카드 UI처럼 밋밋하게 처리하는 것
- 캐릭터/전투/카드 핵심 비주얼을 텍스트와 색상 박스만으로 처리하는 것

## 4. 기능 계획

### 4-1. 배포 필수 기능 안정화

P0:

- Firebase Auth: 이메일/Google 로그인, 로그아웃, 신규/기존 사용자 라우팅
- Firestore 저장/복원: 캐릭터, 퀘스트, 인벤토리, 던전 진행도
- 알림: Android 13+ 권한, 예약/취소, 타임존
- AdMob: 운영 App ID/Ad Unit ID 확인, 테스트 모드 분기 확인
- IAP: 상품 조회, 구매, 복원, 서버 검증, 실패/취소 처리
- Crashlytics: FlutterError, 비동기 에러, Firebase 초기화 실패 기록
- 홈 위젯: 위젯 갱신 실패 시 앱 크래시 없음

P1:

- 오프라인 모드: 네트워크 복구 후 동기화
- 설정: 언어/테마/사운드/알림/개인정보 링크
- 데이터 삭제/계정 삭제 흐름
- 사운드 설정 저장 및 전역 반영

완료 기준:

- 기능별 수동 QA 체크리스트 작성
- 실패 케이스가 UI에 표시되고 앱이 멈추지 않음
- Crashlytics에서 출시 전 테스트 크래시 수집 확인

### 4-2. Soul Deck 기능 안정화

P0:

- 전투 시작/종료/승리/패배/보상 루프
- 카드 사용 비용, 대상 선택, 다중 적, 상태이상, 유물 효과
- 덱 셔플, 드로우, 버림 더미, 소멸, 업그레이드
- 전투 중 앱 백그라운드/복귀 안정성
- 카드 보상 선택/스킵, 상점 구매, 휴식 업그레이드

P1:

- 전투 로그 또는 최근 행동 표시
- 카드 키워드 도움말
- 첫 전투 튜토리얼
- 전투 사운드/햅틱 피드백

완료 기준:

- 최소 20회 연속 전투 자동/수동 시나리오에서 크래시 없음
- 5개 존 각 1회 클리어 루프 확인
- 무한 타워 30층 이상 진행 확인

## 5. QA 계획

### 5-1. 자동 검증

기본 명령:

```bash
flutter analyze
flutter test
flutter build appbundle --release
```

추가 권장:

- 카드 데이터 무결성 테스트: 업그레이드 ID, 비용 범위, 효과 파라미터, 로컬라이제이션 키 존재
- 몬스터 데이터 무결성 테스트: spritePath 존재, HP/공격 범위, 챕터 보스 배치
- 에셋 경로 테스트: `pubspec.yaml`에 선언된 디렉터리와 실제 파일 매칭
- ARB 키 테스트: 4개 언어 키 동기화
- 저장/복원 회귀 테스트: CharacterState, DungeonState, PurchaseService, AdService

### 5-2. 실제 기기 QA

필수 기기:

- 저사양 Android 8~10
- 중급 Android 12~13
- 최신 Android 14~15 또는 16 대응 기기
- 작은 화면 360dp 폭
- 큰 화면/태블릿 가능하면 1대

테스트:

1. 최초 설치부터 온보딩 완료
2. 로그인/로그아웃/재로그인
3. 퀘스트 생성/완료/삭제/반복
4. 전투 5회, 보스 1회, 패배 1회, 보상 선택 3회
5. 광고 시청 성공/실패/일일 제한
6. IAP 테스트 상품 구매/복원
7. 알림 예약 후 앱 종료 상태 수신
8. 네트워크 끊김/복구
9. 앱 강제 종료 후 데이터 복원
10. 30분 연속 사용 중 발열/프레임/메모리 관찰

### 5-3. 출시 판정 게이트

정식 배포 가능:

- P0 버그 0개
- `flutter analyze` 0 issue
- `flutter test` 전체 통과
- release AAB 생성 성공
- 실제 기기 최소 3대에서 핵심 루프 완료
- Play Console 필수 항목 완료
- 개인정보처리방침 URL 활성
- Crashlytics 테스트 이벤트 수집 확인

정식 배포 금지:

- 로그인/저장/구매/광고/전투 중 하나라도 실제 기기에서 재현 가능한 크래시가 있음
- 카드/전투 핵심 화면이 도형 임시 비주얼 상태로 남아 있음
- AdMob/Firebase/IAP 운영 설정이 확인되지 않음
- 앱 삭제/계정/개인정보 관련 사용자 고지 누락
